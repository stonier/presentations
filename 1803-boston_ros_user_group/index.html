NaviStack

Great to get Started, but there is no Silver Bullet

Walkthrough
  • Navistack is a 'Point to Point' Navigation System
  • Originally seems to be a Controller/Planner
  • Planners are nicely pluggable
  • Move Base is really quite trivial
  • Navi Layers at first seem nicely pluggable, but
  • Navi Layers are not probabilistic
  • Navi Layers enforce non-programmatic constraints on the fusion
 
lidar_layer:
  enabled:               true
  footprint_clearing_enabled: true
  ...
  combination_method:    1
  accuracy_multiplier_bits: 20
  clear_corner_cases:    true
  track_unknown_space:   true
  use_cached_updating:   true
  observation_sources:   left right front_left front_right
  extensions:
    do_selective_clearing: false
    reset_every_cycle:     false

NaviStack

Start thinking about Navigation...

Many Contexts
All require Perception-Planning-Control

  • Navigate from A to B <- Move Base
  • Navigate along this corridor
  • Park/Unpark
  • Dock/Undock
  • Initialise the localisation
  • Elevator entry/exit
  • Traverse this ramp

Keep routing separate! (i.e. don't make the 'super planner')
Keep it simple! (e.g. don't add 'sounds', do that elsewhere)

NaviStack

Ready for your Own NaviStack?


  • Listen to your use case - there is no silver bullet (yet) and this needs to be 100% robust
  • Grid Map is likely to be at the core of it
  • Routing at a much higher level (often used in a cost analysis before tasking a robot)
  • Stitch contexts / notifications / exception handling in a decision engine (e.g. behaviour trees)

Borked Agile

What Worked for Our Situation

Divide teams along lines of development overlap
(localisation, robot control + trees, server + web)
Drop complicated 'JIRA' like software
Hide irrelevant noise and planning from each group
BizDev & Service Design → Broad requirements & Constraints
Teams → Architectural reqiurements & Constraints
Sequence Epics in a Dependency Graph
Bump priority for design of Epics of Impending Doom
Weekly Snapshots over a google doc with each team (aka sprint)
Requirements → Design → Implement → PR → Code Review → Iterate → Docs Cycle
Focus on clarity for Devs
Focus on assisting Devs with their weak points
Focus on being the crystal bridge between teams
Not recommendations for every situation, but our situtation
Be clear about who was doing the epics...getting it out of the way of devs
Interesting, once we had something working (third try), got good feedback from the devs
